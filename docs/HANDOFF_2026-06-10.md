## 3. VWAP staleness guard (shipped — verify live)
Root cause of false `vwap_rise` churn on illiquid CRUDEOIL: a quiet leg's broker ATP is forward-
filled/frozen; a stale single-leg ATP could set `session_min_vwap` low → later normal reads "rise"
above it. Fix:
- `PoolIndicatorEngine`: `self._last_atp_ts[key]` stamped **only on a real atp>0 tick** (kept-last-
  good 0 does NOT refresh). New `pair_atp_fresh(ce, pe, max_sec)`.
- `sell_straddle.py` vwap_rise block (~L1390): the WHOLE step is skipped (no `session_min_vwap`
  read/update) when `pair_atp_fresh(ce, pe, self._vwap_stale_sec)` is False. `_vwap_stale_sec`
  default 90s, per-index overridable via `vwap_rise_sl.stale_sec` / `ss.vwap_stale_sec`.
- EXIT-EVAL log: tf=1 `stale` flag + `VWAPrise(... STALE-skip)` so it's diagnosable.
- **Verify on live CRUDEOIL (MCX to ~23:30):** see `stale=1.0`/`STALE-skip` during quiet PE minutes,
  no vwap_rise firing off a frozen leg; NIFTY still rolls normally with both legs live.
